## 一、SVM 训练——"教会计算机认字"

### 1.1 类比：教小孩认字

你教小孩认"京"字：
1. 给他看 80 张"京"的图片（不同字体、有点脏、有点歪）
2. 每张告诉他"这是京"
3. 小孩自己总结规律
4. 下次看到新的"京"字，能认出来

SVM 训练跟这个一模一样：
1. 用 4 种字体渲染 65 类字符，再加噪声/旋转（数据增强）
2. 每张图片提取 HOG 特征（轮廓形状的数字描述）
3. SVM 学习"什么样的特征对应什么字"
4. 新字符来了，提取 HOG → SVM 判读 → 返回字符+置信度

### 1.2 HOG 特征是什么

把一张 32×32 的图片切成 4×4 的小格子（cell），每个格子里统计：
- 这个区域边缘的方向分布（横的/竖的/斜的）
- 9 个方向区间 × 4×4 cells × 4 blocks = 324 个数字

这 324 个数字就是该字符的"指纹"。

类比：你描述一个人的脸——眼睛多大、鼻子多高、嘴巴多宽。把这些数字放一起就是 HOG。

### 1.3 为什么要合成数据

真实车牌数据集（CCPD）几百 GB。对于理解原理 + 面试展示，合成数据完全够用：

- 4 种中文字体覆盖了常见车牌字体风格
- 噪声+旋转模拟了真实拍摄的模糊和倾斜
- 99.52% 的准确率证明了方法的有效性

如果面试官问"为什么不用真实数据"，回答：合成数据可控、可复现、不依赖外部数据集下载。生产环境可以替换为真实数据训练，接口不变。

## 二、重试机制——"API 挂了也别怕"

### 2.1 类比：打电话

你给朋友打电话：
1. 第一次没人接 → 等一会儿再打（重试）
2. 第二次还是没人接 → 再等更久（指数退避）
3. 第三次还是没人接 → 发微信留言（降级）

但如果电话里说"您拨打的号码是空号"（API key 错误），你不会再打第三次——打了也没用。

### 2.2 三层保护

```
第 1 层：超时保护
  → 30 秒没回复就算失败，不无限等待

第 2 层：智能重试
  → 网络波动 → 重试（指数退避 1s → 2s → 4s）
  → API key 错误 → 不重试，直接降级

第 3 层：降级兜底
  → 3 次全失败 → 用 SVM 结果，不阻塞流水线
```

**关键设计**：降级不阻塞流水线。LLM 复核是"锦上添花"，不是"没有它就不行"。SVM 结果虽然不够准，但至少有——不会因为 DeepSeek 挂了就让用户看到"?"。

## 三、Parallel 并行——"7 个人同时干活"

### 3.1 类比：餐厅后厨

串行：一个厨师做 7 道菜，做完一道再做下一道 → 慢

并行：7 个厨师同时做 7 道菜 → 快

车牌识别的 7 个字符互不依赖——第 2 个字符的识别不依赖第 1 个的结果。天然可以并行。

### 3.2 怎么实现

Python 里 `tool_svm_predict` 是同步函数（不涉及网络 IO），用 `ThreadPoolExecutor` 把它扔到线程池里并发跑：

```python
executor = ThreadPoolExecutor(max_workers=8)
results = list(executor.map(tool_svm_predict, char_paths))
# 7 个字符同时跑 → 6ms，而不是 7 × 160ms = 1130ms
```

在 GraphAgent 节点里，用 `asyncio.gather` 实现异步并行：

```python
loop = asyncio.get_event_loop()
tasks = [loop.run_in_executor(_svm_executor, tool_svm_predict, p)
         for p in char_images]
results = await asyncio.gather(*tasks)
```

### 3.3 效果

```
Sequential: 1130.5ms
Parallel:     6.1ms
Speedup:    184.8x
```

185 倍加速——不是因为并行算法多厉害，是因为 7 个字符 7 个线程同时算，总耗时 ≈ 最慢的那个字符的耗时。

## 四、Interrupt / Resume——"暂停等我确认"

### 4.1 类比：快递签收

快递到付 → 快递员打电话"你在家吗？" → 你说"在" → 他送上来。

这就是 interrupt / resume：
- 快递员暂停送件流程 = interrupt
- 你回复"在"之后他继续 = resume

在 GraphAgent 里：
- 识别到极低置信度字符（conf < 0.5）→ interrupt
- 把"请确认这些字符"发给用户
- 用户回复确认 → resume，继续往后走

### 4.2 tRPC-Agent 的 interrupt API

```python
from trpc_agent_sdk.dsl.graph import interrupt

async def human_review_node(state, writer):
    # 暂停执行，返回确认请求给客户端
    human_response = interrupt({
        "message": "字符 '5' 置信度 0.32，请确认",
        "options": ["确认", "改为 S"]
    })
    # resume 后，human_response = 用户的选择
    return {"confirmed": human_response}
```

### 4.3 Checkpoint——"中断时的存档点"

游戏打到一半→存档→退出→下次从存档继续。

GraphAgent 的 checkpoint 就是存档：
- interrupt 前自动把当前 state 存下来
- resume 时从 checkpoint 恢复 state
- 不管中断多久，恢复后跟没中断一样

这是 LangGraph 提供的底层能力，tRPC-Agent 通过 `interrupt()` 封装暴露。

## 五、防幻觉四道闸门——完整回顾

```
用户上传车牌图片
  │
  ▼
预处理 → 定位 → 分割
  │
  ▼
Parallel SVM 识别（7字符并行）
  │
  ├── conf ≥ 0.85 ──────────────→ 第 1 道闸门：高置信度直接通过
  │
  ├── 0.5 ≤ conf < 0.85
  │     │
  │     ├── LLM 复核（RAG 混淆字符）
  │     │     ├── retry 3 次 ──→ 第 2 道闸门：网络波动自愈
  │     │     └── 全部失败 → SVM 兜底 ──→ 第 3 道闸门：降级不丢字符
  │     │
  │     └── 输出结果
  │
  └── conf < 0.5
        │
        ├── interrupt 人工确认 ──→ 第 4 道闸门：极低置信不自动输出
        └── 用户确认后 → 输出结果
```

**考试时的口述**：我构建了四层防幻觉体系——SVM 置信度阈值过滤高置信度字符，LLM 复核处理低置信度（含 RAG 混淆字符检索和三层容错），极低置信度触发人工确认（interrupt 模式），最终输出标注低置信度字符让用户知情。

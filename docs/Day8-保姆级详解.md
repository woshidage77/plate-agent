## 零、Day 7 的评测差在哪里

Day 7 的评测：predicted == ground_truth 才算对。太严格了。

真实场景：SVM 把 "5" 认成 "S" → 精确匹配判 0 分。但用户只差一个字符。

Day 8 的 LLM Judge：**让 DeepSeek 当裁判，用"人"的标准打分**。

---

## 一、Judge 怎么打分——考试阅卷类比

- 精确匹配 = 机器阅卷（答案必须一字不差）
- LLM Judge = 老师阅卷（允许步骤对但结果有小误差）

| 识别情况 | 精确匹配 | LLM Judge |
|---------|---------|----------|
| 京A12345 vs 京A12345 | 1.0 | 1.0 |
| 京A12345 vs 京A1234S | 0.0 | 0.8 |
| 京A12345 vs ?????? | 0.0 | 0.0 |

---

## 二、核心代码——prompt 就是灵魂

eval/judge.py 的识别质量 prompt：

```python
RECOGNITION_PROMPT = """你是一个车牌识别系统的评测裁判。

识别目标（ground truth）：{ground_truth}
系统预测（predicted）：{predicted}

评分标准：
- 1.0：完全正确
- 0.8：1个字符错误（常见 OCR 混淆，如 B/8、0/O、S/5）
- 0.6：2个字符错误
- 0.4：3个字符错误
- 0.2：仅省份或首位正确
- 0.0：完全错误或识别失败

请以 JSON 格式回复：
{"score": 0.8, "reason": "..."}"""
```

关键设计：
- **评分标准写死在 prompt 里**——不是让 LLM 自由发挥
- **常见混淆字符列出来**——B/8、0/O、S/5 是高频错误
- **JSON 约束 + temperature=0**——确保可解析 + 一致性

---

## 三、三维评分 vs 单一分数

Day 7：一个分数（对了/错了）。
Day 8：三维——识别质量 + 黑名单质量 + 回复质量。

改进了预处理代码后，即使识别准确率不变，回复质量可能提升——Judge 能捕捉到精确匹配看不到的改进。

---

## 四、验证结果解读

2 张样本 + Judge：

```
识别质量均值: 0.00 ← SVM 返回 "??????"，正确判 0 分
黑名单质量均值: 0.00
回复质量均值: 0.50 ← 至少跑了流水线并返回了结果
```

当 SVM 训练完成后，三个数字一起涨——Judge 的价值。

## 五、考试速记卡

| 考点 | 答案 |
|------|------|
| LLM Judge 做什么？ | 用 LLM 评估系统输出质量，允许合理误差 |
| 三个评测维度？ | recognition / blacklist / response quality |
| 为什么 temperature=0？ | 确保评测一致性 |
| 评分标准写在哪？ | Prompt 里，作为 rubric 明确列出 |
| Judge 评分范围？ | 0.0~1.0，越高越好 |

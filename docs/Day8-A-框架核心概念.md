# PlateAgent Day 8-A：LLM Judge 评测体系（考试向）

> 源：LLM-as-Judge 论文方法论 + 项目实战
> 用途：7.1 犀牛鸟考试知识储备

## 一、为什么需要 LLM Judge

### 1.1 精确匹配评测的局限

Day 7 的评测：predicted == ground_truth 才算对。

问题：SVM 把 "5" 误认为 "S" → 精确匹配判 0 分。但从用户角度看，"京A1234S" 和 "京A12345" 只差一个字符——结果是"基本可用"的。

### 1.2 LLM Judge 的思路

让 LLM 当裁判，评估输出质量而非简单比对。

## 二、三个评测维度

| 维度 | 评估内容 | 评分范围 |
|------|---------|----------|
| recognition_quality | 识别结果是否合理（允许 OCR 误差） | 0~1 |
| blacklist_quality | 黑名单查询是否恰当 | 0~1 |
| response_quality | 整体回复是否清晰完整 | 0~1 |

### 识别质量评分标准

| 分数 | 含义 |
|------|------|
| 1.0 | 完全正确 |
| 0.8 | 1字符错误（常见 OCR 混淆） |
| 0.6 | 2字符错误 |
| 0.4 | 3字符错误 |
| 0.2 | 仅省份或首位正确 |
| 0.0 | 完全错误或识别失败 |

## 三、LLM Judge 的设计原则

- **结构化 Prompt**：评分标准写死在系统提示里
- **零温度**：temperature=0.0 确保评测一致性
- **JSON 输出约束**：强制 LLM 返回可解析的 JSON

## 四、与精确匹配互补

| | 精确匹配（Day 7） | LLM Judge（Day 8） |
|---|---|---|
| 优点 | 客观、确定、快速 | 理解语义、容忍误差 |
| 缺点 | 过于严格 | 有 API 调用开销 |
| 适用 | 字符级监督 | 用户体验评估 |

## 五、考试速记卡

| 考点 | 答案 |
|------|------|
| LLM Judge 做什么？ | 用 LLM 评估系统输出质量 |
| 三个评测维度？ | recognition / blacklist / response quality |
| 为什么 temperature=0？ | 确保评测一致性 |
| 与精确匹配的关系？ | 互补——一个严格，一个灵活 |
